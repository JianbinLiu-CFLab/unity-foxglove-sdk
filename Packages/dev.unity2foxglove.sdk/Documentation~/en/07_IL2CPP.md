# 1. IL2CPP Build Guide

IL2CPP (Intermediate Language To C++) is the standard backend for Unity standalone platform builds. This guide covers the key points for building the Unity2Foxglove SDK as an IL2CPP Player.

## 1.1 Purpose

Use this guide to build and verify Unity2Foxglove in an IL2CPP Player.

## 1.2 Application

Read this before release builds, CI builds, or any test where Editor-only Roslyn/source-generation behavior may differ from Player behavior.

## 1.3 Why IL2CPP matters

- Unity standalone platform builds (Windows/Linux/macOS) use IL2CPP by default
- IL2CPP converts C# IL to C++ and compiles it to native code during the build
- Unlike the Mono backend, IL2CPP performs code stripping, which may remove types used by reflection
- IL2CPP does not support runtime Roslyn ISG, requiring a fallback scheme to generate source code

## 1.2 link.xml preservation

IL2CPP code stripping may remove types that the SDK depends on. A `link.xml` must be present in the project.

The SDK includes `Runtime/link.xml` in the package:

```xml
<linker>
  <!-- Preserve Newtonsoft.Json -- heavy reflection usage -->
  <assembly fullname="Newtonsoft.Json" preserve="all" />
  <!-- Preserve Foxglove SDK runtime -->
  <assembly fullname="Unity.FoxgloveSDK" preserve="all" />
</linker>
```

Unity automatically collects `link.xml` from all assemblies during IL2CPP builds. If your project has another link.xml, ensure there are no conflicts.

## 1.3 [FoxRun] source generation

### 1.3.1 Problem

`[FoxRun]` relies on the Roslyn Incremental Source Generator (ISG) to generate `_FoxRun.g.cs` during Editor compilation. However, during IL2CPP builds:
- The Roslyn ISG DLL does not run in the IL2CPP compilation pipeline
- Therefore, the `IFoxgloveLogSource` implementation code does not exist

### 1.3.2 Solution: two-stage generation

#### 1.3.2.1 Stage 1: Editor mode (ISG)

In the Unity Editor, `FoxgloveLogSourceGenerator` (Roslyn ISG) runs automatically. Generated code exists in compilation memory with no physical files.

DLL location: `Editor/SourceGenerators/analyzers/dotnet/cs/FoxgloveLogSourceGenerator.dll`

#### 1.3.2.2 Stage 2: Pre-build fallback (IPreprocessBuildWithReport)

Before the IL2CPP build starts, `FoxrunBuildPreprocess` runs automatically:

1. Scans all assemblies for partial classes with `[FoxRun]`
2. Calls `FoxrunCodeGenerator.GenerateSourceFiles()` to generate real `.g.cs` files
3. Outputs files to the `Assets/Scripts/Generated/` directory
4. Calls `AssetDatabase.Refresh(ForceSynchronousImport)` so Unity compiles the new files
5. Generated code is wrapped in `#if !UNITY_EDITOR`, participating only during Player compilation

Example Console log:

```
[FoxrunBuildPreprocess] Generating FoxRun source files...
[FoxrunCodeGenerator] Generated TestLog_FoxRun.g.cs
[FoxrunBuildPreprocess] Generated 1 file(s): TestLog_FoxRun.g.cs
```

### 1.3.3 Generated file format

```csharp
// <auto-generated/>
// Generated as a physical fallback for Player/IL2CPP builds.
#if !UNITY_EDITOR
#pragma warning disable
using UnityEngine;
using UnityEngine.Scripting;
using Unity.FoxgloveSDK.Components;

partial class TestLog : IFoxgloveLogSource
{
    int IFoxgloveLogSource.FoxgloveLog_TopicCount => 2;

    FoxgloveLogTopicInfo IFoxgloveLogSource.FoxgloveLog_GetTopic(int index) { /* ... */ }

    [Preserve]
    void IFoxgloveLogSource.FoxgloveLog_Publish(int topicIndex, FoxgloveManager mgr, ulong nowNs) { /* ... */ }
}
#endif
```

### 1.3.4 Generated file management

- Files are in `Assets/Scripts/Generated/` and should be added to `.gitignore`
- They are regenerated before each build (only written when content changes, preventing unnecessary recompilation)
- They can be cleaned up manually after building

## 1.4 Build script

The project provides a Python build script `Scripts/build_unity_il2cpp.py`:

```bash
# Build Windows x64 IL2CPP Player
python Scripts/build_unity_il2cpp.py --target win64

# Build Linux x64 IL2CPP Player
python Scripts/build_unity_il2cpp.py --target linux64

# Build macOS IL2CPP Player
python Scripts/build_unity_il2cpp.py --target macos --unity /path/to/Unity.app
```

### 1.4.1 Build output

Build files are output to: `build/Unity/<target>-il2cpp-<timestamp>/`

Example: `build/Unity/win64-il2cpp-20260505_143021/`

### 1.4.2 Build log monitoring

The script monitors the Unity build log in real time, highlighting these markers:
- `[Foxrun` -- FoxRun generation logs
- `Build succeeded` / `Build failed` -- build results
- `error CS` -- C# compilation errors
- `Exception` -- exceptions
- `IL2CPP` -- IL2CPP conversion progress

## 1.5 Verifying the IL2CPP Player

After building, verify:

1. Run the built `.exe` (Windows) or the corresponding platform executable
2. After the Player starts, FoxgloveManager automatically starts the WebSocket server
3. Open Foxglove Desktop and connect to `ws://127.0.0.1:8765`
4. Confirm all expected topics appear in the Topics panel
5. Confirm the 3D panel displays scene entities correctly
6. Confirm `[FoxRun]` `/debug/*` topics are publishing normally

## 1.6 Compression DLL platform settings

The compression DLLs under `Runtime/Plugins/compression/` need correct platform settings:

### 1.6.1 IonKiwi.lz4.dll (.meta settings)

- **Include Platforms**: all platforms
- **Exclude Platforms**: WebGL

### 1.6.2 ZstdSharp.dll (.meta settings)

- **Include Platforms**: all platforms
- **Exclude Platforms**: WebGL

The SDK's `.asmdef` files have already been configured with the corresponding platform exclusions.

## 1.7 Common issues

| Problem | Cause | Resolution |
|---------|-------|------------|
| IL2CPP build: `MissingMethodException` | Newtonsoft.Json was stripped | Ensure `link.xml` marks `Newtonsoft.Json` with `preserve="all"` |
| IL2CPP build: type missing | `Unity.FoxgloveSDK` assembly was stripped | Ensure `link.xml` marks `Unity.FoxgloveSDK` with `preserve="all"` |
| `[FoxRun]` topics not publishing in Player | .g.cs was not generated | Check build log for `[FoxrunBuildPreprocess]` output |
| Compression exception | DLL platform settings incorrect | Check `.meta` file platform include/exclude settings |
| No `[FoxrunBuildPreprocess]` in build log | `FoxrunBuildPreprocess` not registered | Confirm `callbackOrder = -100` is before other handlers |
| `FoxrunCodeGenerator` cannot find types | Reflection scan failure | Check assembly loadability; `ReflectionTypeLoadException` is skipped |
